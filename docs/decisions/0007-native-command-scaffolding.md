# 0007 — native-command scaffolding + mise: manifests are process-deterministic task-output

- Status: accepted
- Date: 2026-07-14
- Governs: spec 011 (de-opinionated module family); reconciles Constitution III (v2.3.0).

## Context

Spec 011 re-scopes the `bailiff-mod-*` family from a faithful `project-setup` translation into a
de-opinionated module family. Two ratified decisions from the 2026-07-14 session change how
language/tool modules produce their primary artifacts:

1. **`mise` is the default tool/version manager.** Each language/tool module writes (or
   contributes to) a `.mise.toml` pinning its tools + versions, and its preflight `_task` becomes
   `mise install` (or a mise-aware presence check) instead of per-tool `command -v` checks.
2. **Native tool commands scaffold projects.** Initial manifests are produced by the tool's own
   init — `uv init`, `bun init`, `pnpm init`, `cargo new`, `go mod init`, `cdk init` — run as a
   trust-gated `_task`, rather than being hand-rendered as byte-identical Jinja templates. Adding
   dependencies later (`bailiff-mod-package-add`) uses the tool's native `add` command.

The Phase-0 module (`bailiff-mod-python`) rendered `pyproject.toml` as a Jinja template, so its
reproduce was **byte-identical** — the strong form of Constitution III. Native init changes that:
`uv init` etc. generate the manifest, and the exact bytes can vary with the tool version even when
inputs are fixed. This ADR records the decision to accept that shift and how it stays faithful.

Key verified facts:
- Constitution III already carves out task side-effects: "`_tasks` run at both init and reproduce
  … Reproduce is process-deterministic (same pinned inputs → same commands), not necessarily
  byte-identical in the world, because tasks may touch external state." The existing
  `LICENSE` (via `gh api`), `.gitignore` (via `gitnr`), and `apm.lock.yaml` (via `apm install`)
  outputs are already in this **task-output** category.
- bailiff is focused on **new-project setup**, so requiring the tool to be present at scaffold time
  is acceptable — and `mise` makes it available and version-pinned.

## Decision

1. **Native-tool-generated manifests are task-output**, the same category the constitution already
   defines for LICENSE/gitnr/apm-lock. They are **process-deterministic** (same `mise`-pinned tool
   version + same frozen answers → equivalent output), NOT asserted byte-identical, and NOT protected
   by a byte/drift test. Reproduce re-runs the native init (idempotency-guarded: skip if the manifest
   already exists — the LICENSE-guard pattern), so a fresh-checkout reproduce regenerates the manifest
   and a populated-tree re-run leaves the project-owned file alone (`_skip_if_exists`).

2. **The determinism boundary is drawn per file, not per module:**
   - **task-output** (process-deterministic, native/network task): tool-owned manifests
     (`pyproject.toml`, `package.json`, `Cargo.toml`, `go.mod`, CDK app files), `.gitignore` (gitnr),
     `LICENSE` (gh), `.terraform.lock.hcl` (`terraform/tofu init`), apm lock.
   - **managed** (byte-identical Jinja render — the strong Constitution III form still holds):
     config bailiff owns and the tool does not generate — e.g. `.tflint.hcl`, `.cfnlintrc.yaml`, the
     CI workflow files, ruff config beyond what `uv init` writes, `.mise.toml` itself.
   - **seed-once** (`_skip_if_exists`): living files the project evolves — `AGENTS.md`, tool
     manifests after their initial generation, `README.md`, `justfile`, seeded IaC source.

3. **Pinning is the determinism contract.** `.mise.toml` pins every tool version; the native init
   command is the only source of the manifest bytes; changing a pin is a template minor bump. No
   `jinja2_time`; `today` injected (Constitution V unchanged).

4. **No irreversible action at scaffold time.** Native tasks may init/scaffold/validate but MUST
   NEVER perform a deploy/apply (`cdk bootstrap`/`deploy`, `terraform apply`, `sam deploy`,
   `gh repo create` without the public-repo consent gate). Trust-gating + the per-action safe-default
   booleans (spec 009 Q5) still govern.

## Consequences

- **Positive**: generated projects match what each tool's own `init` produces (authentic, idiomatic,
  maintained by the tool author, not reimplemented in Jinja that drifts from the tool's conventions).
  De-opinionation (FR-014) is simpler — the tool owns the manifest format for its chosen
  package-manager/layout, so bailiff branches far less template logic. Cross-tool correctness (e.g.
  pnpm's `pnpm-workspace.yaml` vs npm's `package.json workspaces[]`) is the tool's responsibility.
- **Negative**: reproduce of a manifest is no longer byte-identical; a tool release could change the
  generated bytes. Mitigated by `mise` version pinning and the idempotency guard (reproduce onto an
  existing tree does not regenerate; only a fresh checkout does, and it uses the pinned tool).
- **Constitution III is amended (v2.3.0)** to name native-tool-generated manifests as an explicit
  task-output member — a MINOR clarification, not a reversal (III already had the process-deterministic
  carve-out). The faithful + agent-free intent is unchanged.
- **Testing**: loop tests stub the native commands to deterministic offline marker writes (as the
  existing gitnr/gh/apm stubs do), so the suite stays hermetic; the byte-identical assertion applies
  only to **managed** renders, and a presence/structure assertion applies to task-output.

## Alternatives considered

- **Render every manifest byte-identically (Phase-0 model)**: keeps strict Constitution III, but
  reimplements each package manager's manifest format in Jinja, drifts from tool conventions, and
  fights the de-opinionation goal (every PM/layout choice becomes a large branch). Rejected per
  maintainer direction.
- **Native init but commit + freeze the exact generated bytes as a managed render**: would restore
  byte-identity but requires capturing tool output at author time and re-rendering it, defeating the
  point of using the tool and going stale immediately. Rejected.
- **No mise (per-tool `command -v` preflight)**: leaves the "is every tool installed at the right
  version" problem unsolved across a multi-language project. Rejected in favor of mise as the single
  pin + install surface.
