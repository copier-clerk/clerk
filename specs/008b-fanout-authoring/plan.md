## Summary

Everything here is **CI bash + template content + config** — no new Python application
code. The deliverables are:

1. `cog.toml` monorepo release config (two mandatory keys + package entries).
2. `_meta/module-template/` — the copier meta-template rendered by `just new-module`.
3. `scripts/check_modules.py` (or shell equivalent) — the `discovery.discover()`-based
   contract lint, callable from pre-commit and cog `pre_bump_hooks`.
4. `scripts/generate_catalog.py` — enumerates modules, reads tags from split repos,
   emits `catalog.json`.
5. `.github/workflows/release.yml` — the one CI job (cog bump → push → detect changed
   → fan-out × N → catalog → gh release).
6. GitHub Actions App token wiring (`actions/create-github-app-token`).
7. `justfile` targets `new-module` and `check-modules`.
8. GitHub Pages config serving `catalog.json`.

## Technical context

**Language**: bash (fan-out job steps), Python 3.11+ (check_modules.py,
generate_catalog.py reusing `src/clerk/discovery.py`).

**New Python deps**: none — check_modules and generate_catalog reuse existing
`discovery.discover()`, `packaging`, `pyyaml`. No new application code; these
are scripts in `scripts/`, not new modules in `src/clerk/`.

**CI platform**: GitHub Actions.

**Auth**: `actions/create-github-app-token` (org App `clerk-fanout`; `contents:write`
+ `administration:write`). App credentials stored as org-level Actions secrets.

**Testing strategy**: Because this spec is CI bash + template content:
- Unit-testable parts: `check_modules.py` (import and call directly with fixture
  dirs); `generate_catalog.py` (mock `git ls-remote`; assert JSON shape).
- CI-only parts: the fan-out workflow and GitHub Pages deploy are integration-only —
  tested against a staging `copier-clerk-staging` org or via a mock `gh` binary.
- The scaffolder is a copier meta-template — tested by rendering it and running
  `check_modules` on the output (must pass immediately).

## Design decisions

### check_modules.py vs shell

Shell could iterate `templates/*/` and call a Python one-liner. A Python script
is cleaner here because it reuses `discovery.discover()` directly (no subprocess
overhead, handles DiscoveryError uniformly) and the three-way parity check reads
both a TOML file (cog.toml) and possibly a separate catalog-sources file. Python
wins on parsability; the script stays in `scripts/` (not `src/clerk/`) because it
is a build-time tool, not application logic.

### generate_catalog.py vs shell

Shell + `jq` could work but is harder to test and fragile on the `git ls-remote`
parsing. A ~60-line Python script is the right call. Reuses `discovery.list_versions()`
for the PEP 440 tag filter (same logic as the consumer plane — C-11).

### Fan-out: shell in the workflow YAML

~25 lines of bash in the workflow is the ADR-0006 decision. A composite action or
a separate reusable workflow are over-engineering for this size. If the fan-out
logic ever needs to be called from outside GitHub Actions, extract it to
`scripts/fanout_module.sh` and call it from the workflow.

### Catalog source registration file

`check-modules` needs to verify three-way parity: `templates/*/` == cog.toml packages
== catalog source entries. The catalog source list must live somewhere the script can
read. Two options:
- Derive from `templates/*/` (zero extra file; parity check becomes "does each
  templates/ dir have a cog.toml entry?" only — no separate catalog list).
- An explicit `catalog-sources.toml` (maintained by the scaffolder; the third leg
  of the parity check).

**Resolved at planning** towards an explicit `catalog-sources.toml` generated by
`just new-module` and verified by `check-modules`. Rationale: makes the catalog
sources a first-class declared artifact, caught early if a module is added without
updating the catalog source. *(If this proves overhead, the script can derive
sources from `templates/*/` and the check reduces to cog.toml parity only.)*

### `pre_bump_hooks` slot

ADR-0006 explicitly reserves `pre_bump_hooks` for preflight checks (pre_bump rolls
back). `check-modules` belongs here — a contract violation before the bump means no
tag is created and no fan-out is attempted.

## Implementation phases

### Phase 1: cocogitto config (unblocked — can be done before 009)

Configure `cog.toml` with the two mandatory settings and an empty `[monorepo.packages]`
section. Wire `check-modules` into `pre_bump_hooks`. This phase needs no modules to
be present — it just sets up the release tooling.

### Phase 2: `_meta/module-template/` scaffolder (unblocked)

Implement the copier meta-template that `just new-module <name>` renders. The
template itself can be authored and tested with a synthetic module name before 009.

### Phase 3: `check_modules.py` linter (unblocked)

Implement the contract lint against `discovery.discover()`. Test with fixture
module directories (synthetic; no real 009 content needed). Wire into pre-commit
and into the justfile.

### Phase 4: `generate_catalog.py` + `catalog.json` (unblocked for scripting; blocked for end-to-end)

The generation script can be written and unit-tested with mock `git ls-remote`
responses before 009. The end-to-end pipeline (real modules → real split repos →
real tags → real catalog) requires 009.

### Phase 5: `.github/workflows/release.yml` + GitHub App (BLOCKED ON 009)

The release workflow can be authored before 009, but it cannot be integration-tested
until at least one real `clerk-mod-*` module exists. The GitHub App (`clerk-fanout`)
must be created and installed in the `copier-clerk` org by a maintainer.

### Phase 6: GitHub Pages hosting (BLOCKED ON 009 end-to-end)

Pages configuration + deployment trigger for `catalog.json`. The infrastructure can
be set up before 009, but the live URL only matters once real modules are published.

## Resolved open questions

- **Q-008b-a (catalog scope)**: modules with no published tag are omitted from
  `catalog.json`; the generator logs them as "not yet released" to stderr.
- **Q-008b-b (Pages trigger)**: triggered on push to main when `catalog.json` changes
  (path filter: `catalog.json`), not on every commit.

## Risks

- **Fan-out bash ownership**: the ~25-line workflow bash is hand-rolled (ADR-0006
  decision). Risk = we own it. Mitigated by: it is a straight `cp → commit → tag →
  push` with a `git ls-remote` idempotency check; no business logic; easy to
  audit. A future maintainer can swap in `danharrin/monorepo-split-github-action`
  (the maintained equivalent) without changing any contract.
- **`git tag --points-at HEAD` fragility** (ADR-0006 note): if the cog bump creates
  the bump commit but HEAD is not exactly that commit (e.g. a force-push race),
  the grep may return stale or empty results. Mitigated by: `git push
  --follow-tags` in step 2 must succeed before step 3 runs; on Actions the runner
  does not change HEAD between steps. If this ever becomes a maintenance burden,
  release-please is the drop-in fallback (ADR-0006).
- **GitHub App setup is a manual one-time step**: the App must be created in the
  org and its App ID + private key stored as org secrets before the workflow can
  run. Document this in the runbook (not a code risk).
