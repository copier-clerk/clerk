# Critique — spec 011 (de-opinionated module family)

**Date**: 2026-07-14 · **Lenses**: product-strategy + engineering-risk (adversarial, code-verified)
**Verdict**: ⚠️ **PROCEED-WITH-UPDATES** — de-opinionation core is sound; fix 4 mechanism-level
defects and decide 1 strategic decoupling before `/speckit.tasks`.

Verification evidence (checked against code, not the spec's self-description): `git tag` →
`clerk-mod-{apm,base,python}-v0.1.0` all released; `src/clerk/ordering.py:11` tie-break =
lexicographic basename; `runner.py:307-352` accumulator = downstream sees upstream *persisted*
answers only; `templates/clerk-mod-base/copier.yml:124-188` — `gitignore_stack` is an
agent-frozen union answer written by a single writer (base), NOT runtime accumulation; constitution
III (line 152) + base `_tasks` (166-208) — tasks run at both init and reproduce.

## MUST-ADDRESS (block a clean task breakdown)

- **M1 — Shared-accreting-file "threading" is unimplementable as written; must be agent-frozen-union
  → single-writer** (cross-cutting §2 mise, §4 hook_manager; quality). With basename tie-break + all
  layers `run_after: [clerk-mod-base]`: `clerk-mod-go` (`…-go`) sorts BEFORE `clerk-mod-precommit`
  (`g`<`p`), so go cannot read precommit's `hook_manager` via the accumulator; and precommit needs
  languages' `hook_blocks` (after languages) while languages need `hook_manager` (after precommit) →
  circular runtime data dependency. `.mise.toml`: base runs first, can't see tokens produced by later
  layers; two managed writers overwrite, not append. **Fix**: rewrite the contract to the real
  `gitignore_stack` pattern — the phase-1 agent freezes a union answer (`mise_tools`, `hook_manager`
  + `hook_blocks`, `quality_languages`) injected as `--data` to a single designated writer; delete the
  "inject at runtime / thread via `default: {{ upstream }}`" framing for accreting files.

- **M2 — No `_migrations`/update story for the RELEASED base/python v0.1.0 → 011 breaking reshape.**
  base moves `.agents/`/`infrastructure/`/`.github/workflows/`/`specs/` out + drops `archive/`/`assets/`;
  python flips `pyproject.toml` from managed render → task-output/seed-once. Both tagged/released;
  Constitution VI mandates the new `_migrations` format; an `update` across managed→seed-once + moved
  dirs is undefined/destructive. **Fix**: declare 011 a non-migratable MAJOR reshape of base/python
  (bump major, no update path) and record why that's acceptable (near-zero consumers), OR author
  `_migrations`. Recommend the former.

- **M3 — Tasks run at reproduce, so `mise install` + native inits fire on every recopy**, hard-coupling
  reproduce to mise + network + still-downloadable pinned toolchains — a reproduce of an OLD project on
  a fresh machine years later can FAIL even for a purely-managed re-render. This is a bigger erosion of
  "faithful reproduce forever" than the byte-identity question the amendment addresses. **Fix**: guard
  preflight/native-init tasks init-only via a committed sentinel (the pattern org-policy already uses),
  so reproduce over a populated tree does not re-shell; and state in FR-019/SC-002 that reproduce
  fidelity is scoped to managed renders.

- **M4 (strategic) — The Constitution III amendment is driven by implementation convenience, bundled
  with (not required by) de-opinionation.** Two separable decisions are fused: (a) offer finite
  `choices:` for tooling axes — the actual user value; (b) generate manifests via native tools + mise.
  De-opinionation (a) is fully achievable on the existing byte-identical render model with
  choice-driven branches (Phase-0 model) — no III amendment, no mise dependency, no reproduce-model
  change. The native-tool authenticity argument is real but is a *separate* tradeoff. **Fix/decision**:
  consider shipping choice-based de-opinionation on the render model FIRST (no III amendment, no SC-008
  release gate), and make native-commands+mise its own follow-on spec. Decouples the risky governance
  change from the safe high-value work.

## RECOMMENDATIONS

- **R1 — Drop `clerk-mod-org-policy` from the 011 build set.** Admitted no-op until an org-source-fetch
  module that doesn't exist; full fan-out cost for a dead module (YAGNI). Ship it in the same spec that
  delivers its only consumer.
- **R2 — Keep apm's empty-set refusal for the specific `install_via_apm && apm_packages==[]` case.**
  Module-level empty (no targets) can be a clean no-op, but silently rendering when the APM install path
  is on with zero packages masks a phase-1 mistake. Two different cases the spec merged.
- **R3 — Ship GitHub CI (all 5 models) first; make GitLab a follow-on.** 10 render paths × rotating pins
  is a maintenance trap and the pin auto-updater (MI-1) is deferred. Don't gate the family on the second
  host; land MI-1 before GitLab.
- **R4 — CI must fail loud when selected with empty `ci_languages` and `monorepo_tool==none`** (currently
  silent no-workflow). Emit a rendered warning or fail preflight.
- **R5 — SC-002 / quickstart §4 assert the wrong thing.** Over the normal (committed-tree) reproduce the
  `test -f` guard + `_skip_if_exists` mean the manifest is used verbatim, byte-identical, NOT regenerated.
  Reword to assert presence/structure on reproduce; what actually changes is init-time cross-machine byte
  consistency, which clerk never strictly guaranteed for task-outputs.
- **R6 — Sequence catalog regen with the apm tombstone** so there's no window where apm is un-discoverable
  in the catalog but not yet redirected by the tombstone.
- **R7 — `clerk-mod-quality` (one-file writer into `.agents/`, now owned by agentic) is a heavy fan-out for
  one token file.** The single-writer rationale is equally met by a `when:`-gated output in precommit or
  agentic. (NOTE: maintainer previously chose to keep it separate — flagged, not re-litigated.)

## QUESTIONS

- **Q-cdk** — `clerk-mod-cdk` is a task-only wrapper around `cdk init` ("template/ renders ~nothing").
  Earned value = trust-gating + never-bootstrap/deploy guard + version pin + cdk.context.json policy.
  Confirm that bundle justifies a separate repo (terraform/cloudformation have real managed content; cdk
  is the thin one).
- **Q-pyver** — `python_version` drives managed outputs (`.mise.toml`, ruff target) AND seed-once
  `requires-python`; after a change + reproduce they diverge. Acceptable, or re-splice `requires-python`
  as a managed edit?
- **Q-goldplating** — `branch_strategy`, `rustfmt_heuristics`, `clippy_stage`, `ui_kit`, `framework`
  (recorded only) are choices whose effect is a recorded string or one line. Confirm each changes output
  materially or demote to documented defaults.
