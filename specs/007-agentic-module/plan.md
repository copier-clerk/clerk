## Summary

Spec 007 ships **one** copier template — `clerk-mod-apm` — that wires an APM
dependency layer into a generated project. Scope was narrowed by the 2026-07-13
clarification (Q1): **v1 is APM only.** MCP config, the SpecKit bridge, and
steering/ADR scaffolding are each deferred to their own future `clerk-mod-*` modules
with their own specs and are NOT part of 007.

The delivery is **pure template + task content** — no new `src/clerk/` module, no new
`scripts/clerk.py` verb (Principle I / C-11). The existing spec-003 ordering engine
and spec-010 invocation surface drive `clerk-mod-apm` identically to any other layer.
The template is authored with the spec-008b authoring tooling already on `main`:
`just new-module name=clerk-mod-apm` scaffolds a contract-complete stub under
`templates/clerk-mod-apm/`, and `scripts/check_modules.py` is the lint the module MUST
pass (answers-file `.jinja`, README, CHANGELOG, three-way registration parity,
published-label immutability).

The module's job (clarified scope):

1. Accept an **APM package set as a runtime-injected list-typed answer** (`--data`,
   ADR-0003) — the phase-1 agent populates it from user input + project requirements,
   the user MAY override, and it **persists to the answers file** so reproduce replays
   the same set (Q2). It is NOT a frozen baked-in `choices:` list.
2. Render a valid **`apm.yml`** into the generated project: the selected package
   entries under `dependencies.apm[]` **plus ≥ 1 APM catalogue/registry source** (a
   sensible default is supplied when the data would yield zero) (Q2 / FR-002a).
3. Install packages via a **trust-gated `_task`** that pins the APM tool version
   (`uv run apm==X.Y.Z …`) for process-determinism (Q3 / FR-004, FR-009).
4. **Refuse** an empty package set rather than render an empty `apm.yml` (Q4).

Reproduce contract (Q3): `apm.yml` re-renders **byte-identically**; `apm.lock.yaml` is
**task-generated external state**, regenerated by the pinned install task under trust
and NOT asserted byte-identical (exactly Constitution III's process-deterministic
model).

Independence (Q5): 007 does **not** depend on spec 009 and is not a prerequisite of
it. `clerk-mod-apm` is built and integration-tested against a **minimal stub base
layer**; ordering is computed at reproduce time by the spec-003 engine from
`when:false` edges. Any real adjacency to a future project-setup module is declared by
*that* module, not baked into 007.

## Technical Context

**Language/Version**: copier template content (YAML + Jinja2). No new Python. Test
glue reuses the existing Python test harness (Python ≥ 3.11, pytest).

**Primary Dependencies**: copier `>=9.16,<10` (already pinned). The install `_task`
invokes the APM CLI via `uv run apm==X.Y.Z` (exact command + apm.yml catalogue-source
key to be confirmed against APM docs — see Residual Open Items).

**Authoring tooling (prerequisite, spec 008b — present on `main`, NOT on this branch)**:
- `_meta/module-template/` + `just new-module name=clerk-mod-apm` — dogfooded copier
  scaffolder that lays down `templates/clerk-mod-apm/` with the answers-file `.jinja`,
  `README.md`, `CHANGELOG.md`, a stub `copier.yml`, and the three-way registration
  edits (`cog.toml [monorepo.packages.clerk-mod-apm]`, `catalog-sources.toml`).
- `scripts/check_modules.py` (`just check-modules`) — the module contract lint the
  finished template MUST pass.
  This tooling landed on `main` via PR #22; the 007 implementation branch rebases onto
  `main` so the tooling is available. If 007 is implemented before that merge point,
  the scaffold task is blocked on it.

**Storage / rendered outputs** (into the generated project's directory):
- `apm.yml` — rendered from the template; committed to the project; **byte-reproducible**.
- `{{ _copier_conf.answers_file }}.jinja` — the answers-file template (required, VI).
- `apm.lock.yaml` — **task side-effect only** (written by the install `_task`); NOT a
  `.jinja` render; **external state**, not byte-guaranteed at reproduce (Q3).
- No MCP / `.specify/` / `.claude/` / `docs/decisions/` outputs in v1 (deferred, Q1).

**Testing**: hermetic, offline. Reuse the spec-003 conftest fixture-builder pattern
(`tests/conftest.py::build_template_repo`, `multi_template_set`) to construct a local
git `clerk-mod-apm` fixture + a **minimal stub base layer** providing the threaded
`project_name`. The network-calling install `_task` is stubbed in fixtures (a no-op
shell command that writes a deterministic marker) so tests stay offline; a real APM
install is out of the hermetic set.

**Target Platform**: developer workstations + CI (macOS/Linux). The `_task` must be a
portable shell command.

**Project Type**: a `clerk-mod-*` copier template in the authoring monorepo under
`templates/clerk-mod-apm/`. No new Python package, no new CLI surface.

**Performance**: none special. Render is fast; the pinned APM install `_task` may take
seconds over the network — acceptable and documented.

**Constraints**:
- Template content only; no new tool code (Principle I / C-01 / C-11).
- APM package set = runtime-injected list answer, persisted (Q2 / ADR-0003).
- `apm.yml` MUST configure ≥ 1 catalogue source; default supplied if none (Q2).
- Empty package set → **refuse**, never render an empty `apm.yml` (Q4).
- All code-executing steps are trust-gated `_tasks` with pinned tool versions (V; Q3).
- Answers-file `.jinja` required; PEP 440 tags; `when:false` edges (VI).
- No `secret: true` questions (VI secrets rule).
- `apm.lock.yaml` is external state — reproduce byte-identity asserted for `apm.yml`
  only (Q3 / Constitution III).

## Residual Open Items (flagged for the implementation step — NOT planning blockers)

These do not block task generation; they are concrete facts to pin during
implementation (a verification task is included in `tasks.md`).

1. **Exact APM install command form.** The pinned `uv run apm==X.Y.Z <install verb>`
   invocation must be confirmed against the APM CLI (does `apm install` read
   `apm.yml` and write `apm.lock.yaml`? what is the exact subcommand?). Recorded in
   the contract once verified. If APM has no CLI install command, the `_task` shape is
   revised accordingly.
2. **`apm.yml` catalogue/registry-source key.** Q2 mandates ≥ 1 catalogue source in
   the rendered `apm.yml`, but the repo's own `apm.yml` expresses package sources
   inline in the `dependencies.apm[]` entries (`gituser/repo/packages/name#ver`) and
   has no standalone `sources:`/`registries:` block. The exact apm.yml key that
   represents a "catalogue/registry source" (and what a sensible default is) must be
   confirmed against APM docs before authoring `apm.yml.jinja`. Until confirmed, the
   requirement is encoded as "≥ 1 source configured, default supplied if none" without
   fixing the key name.
3. **Spec body vs Clarifications drift.** `spec.md` still contains pre-clarification
   material that contradicts the APM-only scope: US3 (select APM + SpecKit, out MCP),
   US4 (steering/ADR), and the US1 acceptance scenario mentioning an "MCP config
   skeleton". The Clarifications section + FR-002/002a/002b/003/005 are authoritative
   (APM only). This plan and tasks follow the Clarifications; the stale US3/US4 and the
   "MCP config skeleton" phrase should be pruned from `spec.md` in a spec-refine pass
   (out of scope for this planning step, which does not rewrite the spec).
